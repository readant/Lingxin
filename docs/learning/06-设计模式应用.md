## 二、设计模式评估

### 2.1 常用设计模式对比

| 模式 | 解决问题 | 复杂度 | 适用场景 | 推荐度 |
|------|----------|--------|----------|--------|
| 模板方法 | 代码重复 | 低 | 多子类有相同流程 | ★★★★★ |
| 策略模式 | 多算法切换 | 中 | 经常切换算法 | ★★★☆☆ |
| 工厂模式 | 对象创建 | 低 | 创建逻辑复杂 | ★★★☆☆ |
| 单例模式 | 全局唯一 | 低 | 全局配置等 | ★★☆☆☆ |
| 观察者模式 | 事件通知 | 中 | 回调机制 | ★★☆☆☆ |

### 2.2 本项目的评估

| 模式 | 是否使用 | 理由 |
|------|----------|------|
| **模板方法** | ✅ 使用 | LSTM 和 Transformer 训练流程完全相同 |
| 策略模式 | ❌ 未使用 | 只有 2 类模型，静态方法足够简单 |
| 工厂模式 | ❌ 未使用 | 模型类型固定，字典映射更直观 |

### 2.3 过度设计的教训

> [!warning] 重要教训
> 设计模式不是银弹！过度设计会增加复杂度，降低可维护性。

**过度设计的标志**：
- 为了"以后扩展"添加大量抽象
- 使用复杂模式解决简单问题
- 创建大量接口和基类
- 项目规模很小但架构很复杂

**本项目的教训**：
最初我引入了策略模式和工厂模式，导致：
- 新增 2 个文件
- 增加约 100 行代码
- 学习成本增加
- 实际收益不大

## 三、模板方法模式详解

### 3.1 模式定义

> **模板方法模式**：定义算法的骨架，将某些步骤延迟到子类中实现。

### 3.2 在本项目的应用

**BaseModel 抽象基类**：

```python
class BaseModel(ABC, nn.Module):
    def train_model(self, train_loader, val_loader=None, epochs=50, lr=0.001):
        """
        模板方法：定义训练算法的骨架
        子类不需要重写此方法
        """
        # 步骤1：准备
        criterion = self._get_criterion()
        optimizer = self._get_optimizer(lr)

        # 步骤2：训练循环
        for epoch in range(epochs):
            self._train_epoch(train_loader, optimizer, criterion)

            # 步骤3：验证
            if val_loader:
                self._validate(val_loader, criterion)

    @abstractmethod
    def forward(self, x):
        """子类必须实现：定义模型结构"""
        pass

    def _get_criterion(self):
        """子类可重写：返回损失函数"""
        return nn.CrossEntropyLoss()
```

**LSTM 子类**：

```python
class LSTMModel(BaseModel):
    def __init__(self, input_shape, num_classes):
        super().__init__()
        self.lstm = nn.LSTM(...)
        self.fc = nn.Linear(...)

    def forward(self, x):
        """只需实现 forward 方法"""
        out, _ = self.lstm(x)
        return self.fc(out[:, -1, :])

    # 继承 train_model，无需重写！
```

## 四、简化方案 - 字典映射

### 4.1 为什么不用工厂模式

对于只有 5 种模型的项目：

```python
# 工厂模式（过度设计）
class ModelFactory:
    @staticmethod
    def get_strategy(model_type):
        strategies = {
            'svm': ClassifierStrategy('svm'),
            'rf': ClassifierStrategy('rf'),
            'lstm': DeepLearningStrategy(LSTMModel),
            ...
        }
        return strategies[model_type]

# 字典映射（简单直接）
MODEL_MAP = {
    'svm': {'type': 'classifier', ...},
    'lstm': {'type': 'deep_learning', ...},
    ...
}
```

**字典映射的优势**：
- 代码更少
- 更容易理解
- 调试更简单
- 性能更好

### 4.2 什么时候需要工厂模式

当满足以下条件时，考虑工厂模式：


## 五、重构经验总结

### 5.1 重构原则

| 原则 | 说明 |
|------|------|
| **YAGNI** | You Aren't Gonna Need It - 不要做过度设计 |
| **KISS** | Keep It Simple, Stupid - 保持简单 |
| **DRY** | Don't Repeat Yourself - 消除重复 |
| **SOLID** | 单一职责、开闭原则、里氏替换等 |

### 5.2 重构检查清单

在引入设计模式前，问自己：

```
□ 1. 真的存在代码重复吗？
□ 2. 重复的代码会影响维护吗？
□ 3. 设计模式会让代码更复杂还是更简单？
□ 4. 这个模式适合当前的复杂度吗？
□ 5. 未来真的需要扩展吗？
□ 6. 团队成员能理解这个模式吗？
```

### 5.3 最佳实践

1. **先写能工作的代码**
   ```python
   # 先让代码工作
   def train_lstm(X, y):
       # 60 行代码
       pass

   def train_transformer(X, y):
       # 60 行代码（重复）
       pass
   ```

2. **识别重复模式**
   ```python
   # 发现两个函数 80% 相同
   # 考虑抽取共同逻辑
   ```

3. **小步重构**
   ```python
   # 每次只重构一小部分
   # 立即测试确保正常工作
   ```

4. **评估收益**
   ```python
   # 重构前：300 行代码
   # 重构后：250 行代码
   # 收益：50 行，复杂度略增
   # 结论：值得重构
   ```

## 七、扩展指南

### 7.1 添加新模型

**步骤 1**：创建模型类

```python
# src/models/gru_model.py
class GRUModel(BaseModel):
    def __init__(self, input_shape, num_classes):
        super().__init__()
        self.gru = nn.GRU(input_shape[1], 128, batch_first=True)
        self.fc = nn.Linear(128, num_classes)

    def forward(self, x):
        _, h = self.gru(x)
        return self.fc(h[-1])
```

